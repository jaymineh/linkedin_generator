## 1. Project Overview

**What it does**: A web app where a user enters a topic, selects a target audience and tone, optionally pastes an article URL, and receives 3 LinkedIn post variations generated by GPT-4o. Users can also import previous posts so the app learns their writing style and can either stay faithful to it or use it as a baseline and improve it. Posts are saved to a history page.

**What makes it AI-native**: The core product value — the posts — is entirely generated by an LLM using a structured output schema. No LLM, no product.

**Two screens**:
- `/` — Generator: input form on the left, 3 generated post cards on the right
- `/history` — History: a searchable list of all previously generated posts

**Style-learning panel on the main screen**:
- import 10-20 previous LinkedIn posts
- generate a reusable writing-style profile
- choose `Off`, `Faithful`, or `Improve` when generating new posts

**API surface (backend)**:
- `POST /api/generate` — takes topic/audience/tone/url, returns 3 posts
- `GET /api/history` — returns paginated list of past generations
- `DELETE /api/history/:id` — deletes a generation
- `GET /health/live` — liveness endpoint (process is up)
- `GET /health/ready` — readiness endpoint (DB reachable, app ready for traffic)

### Capstone Success Criteria

This project is considered complete when these six things are true:

1. **The app works in the container**: the FastAPI backend runs correctly in Docker and serves the API
2. **It is deployed with IaC**: Azure infrastructure is created with Terraform, not by clicking around the portal
3. **It has CI/CD**: GitHub Actions runs checks and deploys on push to `main`
4. **It has observability**: Application Insights, Log Analytics, and Terraform-managed alerts are wired up
5. **It has basic scalability**: Container Apps health probes and autoscaling are configured
6. **It handles secrets securely enough for a capstone**: secrets are passed via GitHub Actions secrets and Container Apps secrets, not hardcoded in code

### What This Guide Intentionally Does Not Try To Be

This is a **capstone deployment guide**, not a blueprint for a huge SaaS product.  
The goal is:

- get the app working in Docker
- deploy it to Azure
- manage infra with Terraform
- automate deploys with GitHub Actions
- add observability and basic scaling

The goal is **not** to build a very large enterprise platform with extra auth layers, complex migration tooling, WAFs, API gateways, custom domains, or multi-service microservice architecture.

## 6. Database Schema

Two tables. For this capstone, keeping schema creation simple in app startup is acceptable and avoids adding a whole migration toolchain.

```sql
-- Stores one row per generation request
CREATE TABLE generations (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    topic       TEXT NOT NULL,
    audience    TEXT NOT NULL,
    tone        TEXT NOT NULL,
    style_mode  TEXT NOT NULL DEFAULT 'off',   -- "off" | "faithful" | "improve"
    url         TEXT,                          -- nullable, article URL if provided
    created_at  TIMESTAMP DEFAULT NOW()
);

-- Stores the individual post variants (3 per generation)
CREATE TABLE posts (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    generation_id   UUID NOT NULL REFERENCES generations(id) ON DELETE CASCADE,
    style           TEXT NOT NULL,             -- "professional", "casual", "storytelling"
    hook            TEXT NOT NULL,             -- opening line
    body            TEXT NOT NULL,             -- full post text
    hashtags        TEXT[],                    -- array of hashtag strings
    created_at      TIMESTAMP DEFAULT NOW()
);

-- Stores the raw imported historical posts used to learn the writer's style
CREATE TABLE style_samples (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    content     TEXT NOT NULL,
    created_at  TIMESTAMP DEFAULT NOW()
);

-- Stores a reusable style profile generated from the imported samples
CREATE TABLE style_profiles (
    id                           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    voice_summary                TEXT NOT NULL,
    opening_patterns             TEXT[] NOT NULL DEFAULT '{}',
    sentence_length_preference   TEXT NOT NULL,
    emoji_usage                  TEXT NOT NULL,
    hashtag_style                TEXT NOT NULL,
    cta_style                    TEXT NOT NULL,
    preferred_topics             TEXT[] NOT NULL DEFAULT '{}',
    phrases_to_mimic             TEXT[] NOT NULL DEFAULT '{}',
    phrases_to_avoid             TEXT[] NOT NULL DEFAULT '{}',
    sample_count                 INTEGER NOT NULL DEFAULT 0,
    created_at                   TIMESTAMP DEFAULT NOW()
);
```

> **Capstone scope note**: this guide intentionally does **not** add Alembic migrations. For a small capstone app with a stable schema, automatic table creation keeps the stack simpler while still letting the app run reliably in Azure.
>
> **Style-learning scope note**: keep this feature single-profile for the capstone. The app stores one active writing-style profile based on the imported posts, rather than introducing auth or multi-profile management.

### `app/database.py`

```python
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, DeclarativeBase
from app.config import settings

engine = create_engine(
    settings.DATABASE_URL,
    pool_size=settings.DB_POOL_SIZE,
    max_overflow=settings.DB_MAX_OVERFLOW,
    pool_pre_ping=True,         # detect stale connections before use
    pool_recycle=300,            # recycle connections every 5 min (Azure closes idle ones)
    connect_args={"sslmode": "require"} if "localhost" not in settings.DATABASE_URL else {},
)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

class Base(DeclarativeBase):
    pass

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

def create_tables():
    from app import models
    Base.metadata.create_all(bind=engine)
```

> **Why pool_pre_ping**: Azure PostgreSQL Flexible Server closes idle connections after a timeout. Without `pool_pre_ping`, the first request after idle will hit a dead connection and return a 500. This pings the connection before use and replaces it if stale.

> **Why sslmode=require**: Azure PostgreSQL enforces SSL by default. Without this, connections from Container Apps will be refused. The conditional skip for localhost keeps local Docker dev simple.

### `app/schemas.py`

Uses `Literal` types so invalid audience/tone values are rejected by Pydantic at the API boundary, before hitting the LLM.

```python
from pydantic import BaseModel, Field
from typing import Literal, Optional
from uuid import UUID
from datetime import datetime

VALID_AUDIENCES = Literal["developers", "executives", "job_seekers", "general"]
VALID_TONES     = Literal["professional", "casual", "storytelling", "thought_leader"]
VALID_STYLE_MODES = Literal["off", "faithful", "improve"]

class GenerateRequest(BaseModel):
    topic: str = Field(..., min_length=3, max_length=1000)
    audience: VALID_AUDIENCES
    tone: VALID_TONES
    style_mode: VALID_STYLE_MODES = "off"
    url: Optional[str] = None

class PostVariant(BaseModel):
    style: str
    hook: str
    body: str
    hashtags: list[str]

class GenerateResponse(BaseModel):
    generation_id: UUID
    posts: list[PostVariant]

class HistoryItem(BaseModel):
    id: UUID
    topic: str
    audience: str
    tone: str
    style_mode: VALID_STYLE_MODES
    created_at: datetime
    posts: list[PostVariant]

    class Config:
        from_attributes = True

class StyleImportRequest(BaseModel):
    posts: list[str] = Field(..., min_length=3, max_length=20)

class StyleProfileResponse(BaseModel):
    voice_summary: str
    opening_patterns: list[str]
    sentence_length_preference: str
    emoji_usage: str
    hashtag_style: str
    cta_style: str
    preferred_topics: list[str]
    phrases_to_mimic: list[str]
    phrases_to_avoid: list[str]
    sample_count: int
    created_at: datetime | None = None
```

> **Why Literal types**: Without these, a user could send `audience: "drop table"` and it would reach the LLM prompt. Literal types enforce allowed values at the schema layer. The `max_length=1000` on topic prevents prompt injection via extremely long inputs.

### `app/middleware.py`

Two production concerns: request tracing and rate limiting.

```python
import time
import uuid
import structlog
from collections import defaultdict
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import JSONResponse
from app.config import settings

logger = structlog.get_logger()

class RequestIdMiddleware(BaseHTTPMiddleware):
    """Adds a unique X-Request-ID header to every request/response for tracing."""

    async def dispatch(self, request: Request, call_next):
        request_id = request.headers.get("X-Request-ID", str(uuid.uuid4()))
        structlog.contextvars.bind_contextvars(request_id=request_id)

        start = time.perf_counter()
        response = await call_next(request)
        elapsed_ms = round((time.perf_counter() - start) * 1000, 1)

        response.headers["X-Request-ID"] = request_id
        logger.info(
            "request_completed",
            method=request.method,
            path=request.url.path,
            status=response.status_code,
            duration_ms=elapsed_ms,
        )
        return response


class RateLimitMiddleware(BaseHTTPMiddleware):
    """
    Simple in-memory sliding-window rate limiter per client IP.
    Only applies to POST /api/generate to protect OpenAI spend.
    """

    def __init__(self, app, rpm: int = 10):
        super().__init__(app)
        self.rpm = rpm
        self.requests: dict[str, list[float]] = defaultdict(list)

    async def dispatch(self, request: Request, call_next):
        if request.method == "POST" and request.url.path == "/api/generate":
            client_ip = request.client.host if request.client else "unknown"
            now = time.time()
            window = [t for t in self.requests[client_ip] if now - t < 60]
            if len(window) >= self.rpm:
                logger.warning("rate_limit_hit", client_ip=client_ip)
                return JSONResponse(
                    status_code=429,
                    content={"detail": "Rate limit exceeded. Try again in a minute."},
                )
            window.append(now)
            self.requests[client_ip] = window

        return await call_next(request)
```

> **Why rate limiting**: Without it, a single user or bot could call `/api/generate` hundreds of times and drain your OpenAI budget. The limit is configurable via the `RATE_LIMIT_RPM` env var.

> **Why request IDs**: Every request gets a UUID that appears in logs and is returned to the client in `X-Request-ID`. When debugging a failed request, the user sends you this ID and you can find the exact trace in Application Insights.

### `app/services/openai_service.py`

This is the core AI logic. Uses OpenAI's structured output (response_format with a Pydantic model) so the response is always valid JSON — no parsing fragility. Includes retry logic for transient failures.

```python
import structlog
from openai import OpenAI, APITimeoutError, RateLimitError, APIConnectionError
from pydantic import BaseModel
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
from app.config import settings

logger = structlog.get_logger()
client = OpenAI(api_key=settings.OPENAI_API_KEY, timeout=30.0)

SYSTEM_PROMPT = """
You are an expert LinkedIn content strategist. Generate exactly 3 LinkedIn post variations.

Each post must:
- Open with a strong hook (the first line determines if someone reads on)
- Be 150–280 words
- Feel authentic, not corporate
- End with a clear thought, question, or call to action
- Include 3–5 relevant hashtags

The 3 styles are always exactly: professional, casual, storytelling.
- professional: data-driven, clear value proposition, authoritative
- casual: conversational, relatable, like talking to a colleague
- storytelling: opens with a personal moment or scenario, narrative arc

Return nothing but the structured JSON.
""".strip()

class PostVariantOutput(BaseModel):
    style: str
    hook: str
    body: str
    hashtags: list[str]

class PostsOutput(BaseModel):
    posts: list[PostVariantOutput]

def build_style_block(style_mode: str, style_profile: dict | None) -> str:
    if style_mode == "off" or not style_profile:
        return ""

    mode_instruction = (
        "Stay as close as possible to this writing style."
        if style_mode == "faithful"
        else "Use this writing style as the baseline, but improve clarity, hook strength, and engagement."
    )

    return f"""

Writing style profile:
- mode: {style_mode}
- instruction: {mode_instruction}
- voice_summary: {style_profile.get("voice_summary")}
- opening_patterns: {", ".join(style_profile.get("opening_patterns", []))}
- sentence_length_preference: {style_profile.get("sentence_length_preference")}
- emoji_usage: {style_profile.get("emoji_usage")}
- hashtag_style: {style_profile.get("hashtag_style")}
- cta_style: {style_profile.get("cta_style")}
- preferred_topics: {", ".join(style_profile.get("preferred_topics", []))}
- phrases_to_mimic: {", ".join(style_profile.get("phrases_to_mimic", []))}
- phrases_to_avoid: {", ".join(style_profile.get("phrases_to_avoid", []))}
""".strip()

@retry(
    retry=retry_if_exception_type((APITimeoutError, RateLimitError, APIConnectionError)),
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=10),
    before_sleep=lambda retry_state: logger.warning(
        "openai_retry",
        attempt=retry_state.attempt_number,
        error=str(retry_state.outcome.exception()),
    ),
)
async def generate_posts(
    topic: str,
    audience: str,
    tone: str,
    style_mode: str = "off",
    style_profile: dict | None = None,
    article_content: str | None = None
) -> list[PostVariantOutput]:

    context_block = ""
    if article_content:
        context_block = f"\n\nReference article (summarise key points, do not copy):\n{article_content[:3000]}"

    style_block = build_style_block(style_mode, style_profile)

    user_message = (
        f"Topic: {topic}\n"
        f"Target audience: {audience}\n"
        f"Preferred tone: {tone}"
        f"{context_block}"
    )
    if style_block:
        user_message += f"\n\n{style_block}"

    logger.info(
        "openai_call_start",
        topic=topic,
        audience=audience,
        tone=tone,
        style_mode=style_mode,
    )

    if style_mode != "off":
        logger.info("generate_with_style_mode", style_mode=style_mode)

    response = client.beta.chat.completions.parse(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user",   "content": user_message},
        ],
        response_format=PostsOutput,
    )

    result = response.choices[0].message.parsed
    logger.info(
        "openai_call_complete",
        posts_generated=len(result.posts),
        prompt_tokens=response.usage.prompt_tokens,
        completion_tokens=response.usage.completion_tokens,
    )
    return result.posts
```

> **Why retries**: OpenAI occasionally returns 429 (rate limit) or 503 (overloaded). Without retry logic, users see a hard failure for a transient issue. The `tenacity` decorator retries up to 3 times with exponential backoff (2s, 4s, 8s) — only for retryable errors, not for 400s or auth failures.

```python
import httpx
import structlog

logger = structlog.get_logger()

async def scrape_url(url: str) -> str | None:
    try:
        async with httpx.AsyncClient(timeout=10, follow_redirects=True) as client:
            response = await client.get(url, headers={"User-Agent": "Mozilla/5.0"})
            response.raise_for_status()
            # Return raw text — the LLM can extract what it needs
            # Strip tags in a basic way (good enough for most articles)
            text = response.text
            # Remove script/style noise with simple slice — not perfect but functional
            return text[:8000]
    except Exception as e:
        logger.warning("scrape_failed", url=url, error=str(e))
        return None  # Scraping failure is non-fatal — generation continues without context
```

### `app/services/style_service.py`

This service turns imported historical posts into a reusable style profile with one structured OpenAI call.

```python
import structlog
from openai import OpenAI
from pydantic import BaseModel
from app.config import settings

logger = structlog.get_logger()
client = OpenAI(api_key=settings.OPENAI_API_KEY, timeout=30.0)

STYLE_ANALYZER_PROMPT = """
You are an expert writing-style analyst.

You will receive a set of LinkedIn posts written by one person.
Analyze them and create a compact style profile that can later be used to generate new posts in the same voice.

Return a profile with:
- voice_summary
- opening_patterns
- sentence_length_preference
- emoji_usage
- hashtag_style
- cta_style
- preferred_topics
- phrases_to_mimic
- phrases_to_avoid

Be specific, but keep the profile compact and reusable.
Return only structured JSON.
""".strip()

class StyleProfileOutput(BaseModel):
    voice_summary: str
    opening_patterns: list[str]
    sentence_length_preference: str
    emoji_usage: str
    hashtag_style: str
    cta_style: str
    preferred_topics: list[str]
    phrases_to_mimic: list[str]
    phrases_to_avoid: list[str]

async def build_style_profile(posts: list[str]) -> StyleProfileOutput:
    joined_posts = "\n\n---\n\n".join(posts)

    logger.info("style_profile_generation_start", sample_count=len(posts))

    response = client.beta.chat.completions.parse(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": STYLE_ANALYZER_PROMPT},
            {"role": "user", "content": joined_posts[:12000]},
        ],
        response_format=StyleProfileOutput,
    )

    profile = response.choices[0].message.parsed
    logger.info(
        "style_profile_generated",
        sample_count=len(posts),
        prompt_tokens=response.usage.prompt_tokens,
        completion_tokens=response.usage.completion_tokens,
    )
    return profile
```

```python
    if request.style_mode != "off":
        stored_profile = db.query(StyleProfile).order_by(StyleProfile.created_at.desc()).first()
        if stored_profile:

                "phrases_to_mimic": stored_profile.phrases_to_mimic,
                "phrases_to_avoid": stored_profile.phrases_to_avoid,

        else:
            logger.warning("style_profile_missing_falling_back_to_off")

    try:
        posts = await generate_posts(
            topic=request.topic,
            audience=request.audience,
            tone=request.tone,
            style_mode=effective_style_mode,
            style_profile=style_profile,
            article_content=article_content,
        )
    except Exception as e:
        logger.error("generation_failed", error=str(e))
        raise HTTPException(status_code=500, detail="Post generation failed. Try again.")

    # Persist to database
    generation = Generation(
        topic=request.topic,
        audience=request.audience,
        tone=request.tone,
        style_mode=effective_style_mode,
        url=request.url,

            body=p.body,
            hashtags=p.hashtags,

    db.refresh(generation)

    return GenerateResponse(
        generation_id=generation.id,
        posts=[{"style": p.style, "hook": p.hook, "body": p.body, "hashtags": p.hashtags} for p in posts],
```

> **Key production improvements over the basic version**:
> - **DATABASE_URL is now a Container Apps secret** — the original had the password in a plain `value` env var visible in the Azure Portal and in `terraform show` output. Now it's stored in the secrets store.
> - **`?sslmode=require` in the connection string** — Azure PostgreSQL enforces SSL. Without this, the backend's connection will be refused.
> - **Liveness + readiness probes** — Container Apps uses these to detect unhealthy containers and stop routing traffic to them. Without probes, a crashed container keeps receiving requests until Azure eventually notices.
> - **HTTP scaling rule** — `concurrent_requests = 20` means Azure adds a replica when each existing replica is handling 20+ concurrent requests. Without a scaling rule, `max_replicas = 5` is never triggered.
> - **Resource tags** — every resource gets `project`, `environment`, and `managed_by` tags. This lets you filter costs, audit ownership, and avoid orphaned resources.
> - **SSL enforced on PostgreSQL server** — `require_secure_transport = on` ensures no unencrypted connections can reach the database.
> - **Backup retention** — `backup_retention_days = 7` is the Azure default but explicitly set for visibility.
> - **Versioned image deploys** — the backend image is no longer hard-coded to `latest`. Terraform now accepts `backend_image_tag`, so every deploy can point to an immutable image like a Git SHA. Without this, a new image push may not trigger a new Container Apps revision.

## 12. Observability

**Section Type**: Must Build

### What you get out of the box

Once `configure_azure_monitor()` is called in `main.py` with a valid connection string, Azure Application Insights automatically captures:

- **Distributed traces**: HTTP request traces with method, path, status code, and duration
- **Dependency tracking**: PostgreSQL queries and outbound OpenAI HTTP calls with timing
- **Exceptions**: Full stack traces for all unhandled errors
- **Custom structured logs**: Every `structlog` event (e.g. `openai_call_start`, `openai_call_complete`, `rate_limit_hit`, `style_import_complete`, `style_profile_generated`) with structured fields
- **Request correlation**: The `X-Request-ID` from `RequestIdMiddleware` appears in all log entries for a given request, so you can trace a single user action end-to-end

### The three observability layers

| Layer | What | Tool |
|---|---|---|
| **Metrics** | Request count, latency histograms, error rates, container CPU/memory | Azure Monitor + App Insights |
| **Logs** | Structured JSON events (OpenAI token usage, rate limit hits, scrape failures) | Log Analytics Workspace (KQL) |
| **Traces** | End-to-end request flow: HTTP → FastAPI → OpenAI → PostgreSQL | App Insights → Transaction Search |

### Key metrics to monitor

| Metric | Where | Why |
|---|---|---|
| `POST /api/generate` latency | App Insights → Performance | OpenAI calls take 3–8s; monitor for spikes |
| `openai_call_complete` log events | Log Analytics → Logs | Track token usage and cost per request |
| `openai_retry` log events | Log Analytics → Logs | Detect if OpenAI is returning transient errors |
| `style_profile_generated` log events | Log Analytics → Logs | Confirm style imports are analyzed successfully |
| `generate_with_style_mode` log events | Log Analytics → Logs | Track when users choose `faithful` or `improve` |
| `rate_limit_hit` log events | Log Analytics → Logs | Identify if legitimate users are being throttled |
| HTTP 500 rate | App Insights → Failures | Catch broken prompts or DB errors |
| Container CPU/memory | Azure Monitor → Container Apps | Detect if replicas need scaling |
| Active PostgreSQL connections | Azure Monitor → PostgreSQL | Detect connection pool exhaustion |
| HTTP 429 response count | App Insights → Performance | Track how often rate limiter fires |

### Alerts (IaC — defined in `alerts.tf`)

Alerts are provisioned via Terraform (see Section 11), not configured manually. Two alerts are pre-defined:

1. **High error rate**: >5 failed requests in 5 minutes → email notification
2. **High latency**: Average response time >15 seconds for 5 minutes → email notification

Enable them by passing `-var="alert_email=you@example.com"` to `terraform apply`.

### Platform logs to retain

For day-2 operations, keep these log categories available in Azure:

- Container stdout/stderr from the backend
- Application Insights traces and dependencies
- PostgreSQL server metrics
- Activity Log events for Terraform-created resources

At a minimum, your runbook should answer:

1. Is the backend up?
2. Is it ready to serve traffic?
3. Are OpenAI calls failing or retrying?
4. Is the DB reachable?
5. Did the latest deploy create a new revision?

### Querying logs with KQL

In Azure Portal → Log Analytics Workspace → Logs:

```kql
-- All generation events in the last hour
traces
| where timestamp > ago(1h)
| where message contains "openai_call_complete"
| project timestamp, message, customDimensions
| order by timestamp desc

-- Average latency of generate endpoint
requests
| where name == "POST /api/generate"
| summarize avg(duration), percentile(duration, 95) by bin(timestamp, 5m)
| render timechart

-- OpenAI retry events (indicates transient failures)
traces
| where timestamp > ago(24h)
| where message contains "openai_retry"
| project timestamp, customDimensions.attempt, customDimensions.error
| order by timestamp desc

-- Style profile generation events
traces
| where timestamp > ago(24h)
| where message contains "style_profile_generated"
| project timestamp, customDimensions.sample_count, customDimensions.prompt_tokens, customDimensions.completion_tokens
| order by timestamp desc

-- Rate limit events by IP
traces
| where timestamp > ago(1h)
| where message contains "rate_limit_hit"
| summarize count() by tostring(customDimensions.client_ip)
| order by count_ desc

-- End-to-end trace for a specific request
traces
| where customDimensions.request_id == "<paste-X-Request-ID-here>"
| order by timestamp asc
```

```yaml
name: Deploy

on:
  push:
    branches: [main]

permissions:
  id-token: write   # Required for Workload Identity Federation
  contents: read

env:
  RESOURCE_GROUP: rg-linkedin-gen-prod

jobs:

    runs-on: ubuntu-latest

      - uses: actions/checkout@v4

      # ── Azure login via Workload Identity (no stored credentials) ──
      - uses: azure/login@v2

          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

      # ── Build & push backend Docker image ──
      - name: Get ACR details
        id: acr
        run: |
          ACR_NAME=$(az acr list -g $RESOURCE_GROUP --query "[0].name" -o tsv)
          echo "name=$ACR_NAME" >> $GITHUB_OUTPUT
          echo "server=$(az acr show -n $ACR_NAME --query loginServer -o tsv)" >> $GITHUB_OUTPUT

      - name: Build and push backend image
        run: |
          az acr build \
            --registry ${{ steps.acr.outputs.name }} \
            --image backend:${{ github.sha }} \
            --image backend:latest \
            ./backend

      # ── Terraform apply ──
      - uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: "1.6.6"

      - name: Terraform init
        working-directory: terraform
        run: |
          terraform init \
            -backend-config="storage_account_name=${{ secrets.TF_BACKEND_STORAGE_ACCOUNT }}"

      - name: Terraform apply
        working-directory: terraform
        run: |
          terraform apply -auto-approve \
            -var="openai_api_key=${{ secrets.OPENAI_API_KEY }}" \
            -var="db_password=${{ secrets.DB_PASSWORD }}" \
            -var="alert_email=${{ secrets.ALERT_EMAIL }}" \
            -var="backend_image_tag=${{ github.sha }}"

      - name: Smoke test backend readiness
        run: |
          BACKEND_URL=$(terraform -chdir=terraform output -raw backend_url)
          curl --fail --silent --show-error "$BACKEND_URL/health/ready"

      # ── Deploy frontend to Static Web App ──
      - name: Get Static Web App deploy token
        id: swa
        run: |
          SWA_NAME=$(az staticwebapp list -g $RESOURCE_GROUP --query "[0].name" -o tsv)
          TOKEN=$(az staticwebapp secrets list -n $SWA_NAME --query "properties.apiKey" -o tsv)
          echo "token=$TOKEN" >> $GITHUB_OUTPUT

      - name: Build frontend
        working-directory: frontend
        run: |
          npm ci
          NEXT_PUBLIC_API_URL=$(terraform -chdir=../terraform output -raw backend_url) npm run build

      - name: Deploy frontend
        uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ steps.swa.outputs.token }}
          action: "upload"
          app_location: "frontend/out"
          skip_app_build: true
```

## 16. Deployment — Step by Step

**Section Type**: Must Build

Follow this exactly for first-time deployment.

### Step 1: Create Terraform remote state storage (once)

```bash
LOCATION="eastus"
RG_NAME="rg-tfstate"
SA_NAME="<unique-storage-account-name>"   # globally unique, alphanumeric, max 24 chars

az group create --name $RG_NAME --location $LOCATION
az storage account create \
  --name $SA_NAME \
  --resource-group $RG_NAME \
  --location $LOCATION \
  --sku Standard_LRS
az storage container create \
  --name tfstate \
  --account-name $SA_NAME
```

### Step 2: Set up GitHub Workload Identity

Run the commands from Section 13 (the `az ad app create` block). Add the 3 output values as GitHub secrets.

### Step 3: Add remaining GitHub secrets

In GitHub → repo → Settings → Secrets → Actions, add:
- `OPENAI_API_KEY`
- `DB_PASSWORD`
- `TF_BACKEND_STORAGE_ACCOUNT`
- `ALERT_EMAIL` (optional)

### Step 4: First Terraform apply (manual, from local machine)

```bash
cd terraform

terraform init \
  -backend-config="storage_account_name=<your-tfstate-storage-account>"

terraform plan \
  -var="openai_api_key=<your-openai-key>" \
  -var="db_password=<strong-password>" \
  -var="alert_email=you@example.com" \
  -var="backend_image_tag=manual-bootstrap"

terraform apply \
  -var="openai_api_key=<your-openai-key>" \
  -var="db_password=<strong-password>" \
  -var="alert_email=you@example.com" \
  -var="backend_image_tag=manual-bootstrap"
```

Note the outputs:
```bash
terraform output backend_url
terraform output frontend_url
terraform output static_web_app_api_key   # needed for GitHub Actions
```

### Step 5: Update CORS after first deploy

Once you know the frontend URL from the output, update `allowed_origins` in Terraform and re-apply:

```hcl
# In your terraform apply, add:
-var="allowed_origins=https://<your-static-web-app-url>"
```

### Step 6: Push to main — GitHub Actions takes over

Every push to `main` after this point will:
1. Build a new Docker image and push it to ACR
2. Run `terraform apply` with the Git SHA as `backend_image_tag`
3. Smoke-test the backend readiness endpoint
4. Build and deploy the Next.js frontend

### Step 6.5: Runbook for a bad deploy

If a production deploy fails:

1. Check the GitHub Actions logs first
2. Check Container Apps revision status in Azure Portal
3. Hit `GET /health/live` and `GET /health/ready`
4. Query Application Insights by `request_id` or failed requests
5. If the backend image is bad, re-run deploy with the last known good `backend_image_tag`

### Step 7: Verify deployment

```bash
# Test the backend health check
curl https://<container-app-fqdn>/health/live
# Expected: {"status":"ok"}

# Test the backend readiness check
curl https://<container-app-fqdn>/health/ready
# Expected: {"status":"ok"}

# Test the generate endpoint
curl -X POST https://<container-app-fqdn>/api/generate \
  -H "Content-Type: application/json" \
  -d '{"topic":"AI in production","audience":"developers","tone":"professional"}'

# Test the style import endpoint
curl -X POST https://<container-app-fqdn>/api/style/import \
  -H "Content-Type: application/json" \
  -d '{"posts":["Post one","Post two","Post three"]}'
```

Open the frontend URL in a browser and generate a post.

### Step 8: Go-live checklist

Before you consider the app productionized, verify all of the following:

- Terraform state is remote and locked in Azure Blob Storage
- CI passes on pull requests
- Deploy workflow uses immutable backend image tags (`backend_image_tag`)
- Container Apps liveness and readiness probes are green
- Azure Monitor alerts are enabled and point to a real email address
- Application Insights is receiving request traces and dependency traces
- `GET /health/live` and `GET /health/ready` both return 200 in production
- CORS only allows your real Static Web App URL
- The frontend can generate posts against the live backend
- Style import works and generates a reusable style profile
- The history page can read and delete generated items
- You know how to roll back to the last known good backend image tag

### Destroying all resources

```bash
cd terraform
terraform destroy \

  -var="backend_image_tag=placeholder"
```

## 17. Cost Estimate

**Section Type**: Optional / Dev Convenience

Running costs when idle (minimal traffic):

| Resource | SKU | Monthly cost (approx) |
|---|---|---|
| PostgreSQL Flexible | B_Standard_B1ms | ~$15 |
| Container Apps | 0.5 vCPU / 1 GB (1 replica) | ~$10–15 |
| Container Registry | Basic | ~$5 |
| Static Web App | Free | $0 |
| Log Analytics | Pay-per-GB (30-day retention) | ~$2–5 |
| Application Insights | Pay-per-GB | ~$1–3 |
| **Total** | | **~$33–43/month** |

To minimise costs while not in use: scale Container Apps to 0 replicas (`min_replicas = 0`) and pause the PostgreSQL server from the Azure Portal.

OpenAI costs: GPT-4o at ~1500 tokens per generation = ~$0.002 per request. For a capstone, this is negligible.
